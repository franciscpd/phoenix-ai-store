# Phase 3: Memory Strategies - Context

**Gathered:** 2026-04-03
**Status:** Ready for planning

<domain>
## Phase Boundary

Deliver memory strategies that transform a conversation's message list before sending to the AI — sliding window, token-aware truncation, pinned messages, summarization, and strategy composition. Strategies do NOT delete messages from storage; they filter what goes in the API call. The full history remains in the database for querying.

</domain>

<decisions>
## Implementation Decisions

### Strategy Behaviour API
- **D-01:** Strategy callback is **always synchronous** — `apply(messages, context, opts) :: {:ok, messages}`. Summarization blocks until completion. Simple and predictable.
- **D-02:** Strategy receives **messages + context** — `apply([Message.t()], context :: map(), opts :: keyword())`. Messages are pure list, context is a separate map with metadata (conversation_id, user_id, model, provider, token limits, etc.). This keeps strategies composable while giving access to metadata when needed.

### Token Counting
- **D-03:** Token counting is **provider-dispatched** — each provider defines how to count tokens via a `TokenCounter` behaviour. Extensible, correct per-provider. Default implementation uses `chars / 4` heuristic.
- **D-04:** Token count is calculated **eagerly on insertion + override** — `token_count` is pre-computed at `add_message` time and stored in the field. Strategies use the stored value by default but can force recalculation via opts (useful when switching models or when `token_count` is nil).

### Pinned Messages
- **D-05:** **System messages are auto-pinned** — any message with `role: :system` is automatically treated as pinned (never evicted by strategies). No manual marking needed for system messages.
- **D-06:** Add **`pinned: boolean` field** to Message struct — allows manual pinning of any message (critical user context, business rules, summaries generated by Summarization strategy).
- **D-07:** Strategies **never delete messages from storage** — they only filter what goes to the AI call. Full conversation history remains in the database for querying. `get_messages/2` returns all messages; strategy output is the filtered subset.

### Composition Model
- **D-08:** Strategies use a **priority-based** composition model — each strategy has a priority, and the system resolves conflicts automatically (e.g., pinned messages always survive regardless of what sliding window or truncation decide).
- **D-09:** Library provides **presets + custom pipelines** — ready-made presets (`:default` = Pinned + SlidingWindow, `:aggressive` = Pinned + TokenTruncation, `:summarize` = Pinned + Summarization + SlidingWindow) plus the ability to define custom strategy chains.

### Summarization Strategy
- **D-10:** Summarization calls the AI synchronously to condense older messages into a summary. The summary message is **pinned automatically** so it survives future strategy applications.
- **D-11:** Summarization is the only strategy with an external side effect (AI call). All other strategies are pure functions on the message list.

### Claude's Discretion
- Exact priority numbers/ordering for built-in strategies
- TokenCounter behaviour default implementation details
- Internal representation of strategy chains
- How presets are defined (module attributes, config, or functions)
- Summarization prompt template

</decisions>

<specifics>
## Specific Ideas

- Strategies are "views" over the message list — they never mutate storage, only filter what the AI sees
- Summarization should generate a summary message with `role: :system` and `pinned: true`, effectively replacing N old messages with one condensed message in the strategy output
- The `context` map passed to strategies should include: `model`, `provider`, `max_tokens` (context window), `conversation_id`, `user_id`

</specifics>
